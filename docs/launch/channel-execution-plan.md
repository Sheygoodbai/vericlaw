# Channel execution plan

Plan date:
`2026-04-05`

This file turns the existing launch copy into an execution sequence for
`VeriClaw 爪印`.

Use it when the goal is not just to publish the repository, but to create real
traffic, replies, stars, forks, downloads, and external citations.

## Primary outcome

For the current GitHub-first launch, the most important outcome is not raw
impressions. It is getting the project in front of the right first operators:

- AI agent builders who already understand bot drift pain
- open-source developers who can star, fork, cite, and test the workflow
- operator-minded users who can describe real failure cases
- Apple-native tool fans who care about the product surface, not only the model

## Conversion ladder

Optimize the launch around this sequence:

1. `See the claim`
   - a person understands in one sentence that VeriClaw is about correction,
     not passive monitoring
2. `Open the repo or landing page`
   - the visitor sees screenshots, proof points, and the public launch surface
3. `Take a strong action`
   - star, fork, download, open the release, or join GitHub Discussions
4. `Create reusable proof`
   - reply, cite, issue, screenshot, fork, or share a correction use case

## KPIs that matter

Do not judge success only by likes.

Track:

- landing page visits
- latest release visits
- GitHub stars
- forks
- release downloads
- GitHub Discussions started
- issues opened with real product feedback
- external posts or citations that mention `VeriClaw 爪印`

## Channel priority

### Tier 1: owned and high-conversion

Start here before broader community posting:

- GitHub release page
- landing page
- README
- GitHub Discussions
- direct founder outreach to warm contacts

Why this comes first:

- these surfaces already match the current public launch posture
- they convert better than broad social reach when the story is still forming
- they create canonical links other people can safely forward

### Tier 2: earned distribution

Use these once the owned surfaces are aligned:

- `Show HN`
- `Product Hunt`
- one AI builder community post
- one Apple-native or indie app community post
- one operator or agent-engineering community post

Inference from current platform rules and the repo posture:

- `Show HN` is strongest when people can actually run, inspect, or try
  something now
- `Product Hunt` is strongest when the visual story is polished and the team is
  ready to answer comments all day
- do not launch both in the same hour because both channels reward active
  participation and follow-up

### Tier 3: social proof loops

After the first wave:

- X / Twitter repost with a screenshot of real feedback
- LinkedIn post with one operator insight, not just the announcement
- friend or partner forwards using the existing templates
- a small shipped update that links back to the original launch

## Recommended 14-day sequence

### Days -7 to -1: prep

- finalize the GitHub release body
- verify the landing page, download page, README, and social preview all match
- prepare one demo walkthrough or screenshot thread
- line up 15 to 30 warm contacts who can give honest feedback or forward the
  launch
- prepare `Product Hunt` assets only if you are ready to monitor comments live
- prepare `Show HN` only if the repo or downloadable build is easy to try

### Day 0: owned launch

- publish the GitHub release
- post the personal founder announcement
- send warm direct messages with a direct ask to read and reply
- open or pin a GitHub Discussion for launch feedback

Success condition for Day 0:

- the first real comments, stars, or replies come from people who understand
  the problem space

### Day 1 or Day 2: `Show HN` if the try path is ready

Use `Show HN` only if you can satisfy the community expectation that people can
meaningfully try the thing, run the repo, or inspect a non-trivial build.

If the try path is still too thin:

- skip `Show HN`
- post a regular link submission later instead of forcing it

### Day 3 or Day 4: `Product Hunt` if the maker workflow is ready

Use `Product Hunt` when:

- the visual story is polished
- the launch page is complete
- the maker first comment is ready
- the team can stay available for replies

If that is not true yet:

- delay it
- do not burn the launch on an underprepared day

### Days 5 to 7: targeted community posts

- post in one AI agent or LLM builder community
- post in one Apple-native or indie app channel
- post in one Chinese forwarding surface you already participate in
- forward the launch kit to relevant partners or operator friends

Rule:

- one tailored post per community
- do not paste the exact same copy everywhere

### Week 2: proof compounding

- quote real feedback
- publish one small improvement or release note
- repost one screenshot, citation, or fork
- ask a satisfied early reader for a short public comment you can link to

## Channel-specific rules

### GitHub release

Use GitHub as the canonical public surface.

Operational notes:

- keep using the `releases/latest` URL in public posts
- attach the strongest screenshots and links in the release
- if Discussions are enabled, create a discussion tied to the release so
  feedback has a durable thread

### `Show HN`

Current official guidance suggests:

- it should be something you made personally
- other people should be able to try it, run it, or inspect it in a meaningful
  way
- avoid barriers like forced signups if possible
- do not ask friends to upvote or comment

Implication for VeriClaw:

- the best `Show HN` version points to a repo or runnable path, not just a
  marketing page
- if the launch is mostly positioning material on that day, wait

### `Product Hunt`

Current official guidance suggests:

- makers can and should self-hunt
- personal accounts are required for posting and participation
- the default planned launch time is `12:01am Pacific Time`
- the maker first comment matters materially
- you can schedule a launch ahead of time, including up to one month ahead
- do not directly ask people to upvote

Implication for VeriClaw:

- use the landing page as the primary URL if it is the cleanest explanation
- use the GitHub release as a strong supporting link in the maker comment
- prepare one clean demo asset and one candid first comment about why the
  project exists
- use [product-hunt-launch-pack.md](product-hunt-launch-pack.md) as the actual
  submission worksheet

### Direct outreach

This is the highest-signal early channel.

Send direct messages to:

- operator friends
- open-source maintainers in adjacent tooling
- Apple-native builders
- anyone who already talked with you about agent reliability

The ask should be honest:

- read it
- tell me if the positioning is clear
- forward it if it genuinely fits someone you know
- use [founder-outreach-playbook.md](founder-outreach-playbook.md) for the
  first `15` to `30` outreach conversations

## Audience-to-message map

### AI agent builders

Lead with:

- `correction, not just monitoring`
- `evidence -> diagnosis -> intervention -> verification`

CTA:

- open the repo
- inspect the workflow
- star or fork if the loop feels useful

### Operator or reliability-minded users

Lead with:

- `catch drift before it compounds`
- `open a case, not just a log`

CTA:

- open the landing page
- review screenshots
- reply with a real failure mode

### Apple-native tool audience

Lead with:

- `Apple-native correction companion`
- `hover and mobile companion surface`

CTA:

- open the landing page
- share the visual launch kit

## Anti-spam rules

Do not:

- ask for fake testimonials
- buy upvotes or comments
- post identical copy across every community
- imply the App Store launch is already live
- overclaim that VeriClaw replaces every observability workflow
